ABSTRACT

This block is part of the Practical Series Automation Library (PAL); a library of software modules and templates that have been made available for the Siemens Simatic S7-1500 range of controllers. It is fully deployable software that is suitable for the control of most industrial process applications.

All the blocks within the PAL conform to the PAL style guide. This block (FC 00001) is a template module that holds common arrangements for various aspects of the block notation and documentation styles.

It sets out the basic approach to documenting a block and includes:

  • Block title and a standard PAL block header

  • Example block descriptions and styling information

  • Revision and modification history examples

  • Typical title, subtitle and text conventions for the PAList

  • Examples usage of figures and equations

  • A list of common symbols and spacing arrangements

The PAL is configured and deployed using the Siemens Simatic TIA Portal programming environment.

The Practical Series Automation Library is freely available under the MIT Open Source licence.

This block is part of the Practical Series Automation Library (PAL). This is a library of software modules and templates that have been made available for the Siemens Simatic S7-1500 range of controllers (and to a lesser extent the S7-1200 range), what used to be referred to as Programmable Logic Controllers or PLCs.

The PAL is configured and deployed using the Siemens Simatic TIA Portal programming environment.

The library is freely available to anyone who wants it — it is available under the MIT Open Source licence. Those who find it useful may, if they wish, make a donation to support the library. Donations can be made here.

The PAL contains fully deployable software that has been developed by the author in his professional capacity as a chartered electrical engineer. It is currently in use on various live plants throughout the UK and in some other parts of the world.

This software is suitable for most industrial applications (typical process applications). It is easy to use and configure, but does have a degree of practical complexity appropriate for the environments within which it is employed. It is heavily configurable, has various operating modes and is suitable for a multitude of industrial applications.

3.1

A note on TIA Portal comment fields

TIA Portal supports comment fields within blocks and within individual networks within a block.

These comment fields are not particularly sophisticated and cannot be customised; they use a fixed font at non-adjustable point size. The font in question is Siemens TIA Portal Basic and it uses a fixed point size of 9 pt that cannot be changed.

The font is a proportional sans-serif font, it looks like this (it is pretty awful):

There are two main problems with this font, the first is the fixed point size is quite small (and it cannot be made bigger), the second is the fact that it is a proportional font.

The older programming package, Simatic Manager (the forerunner to TIA Portal), had a similar comment arrangement; however, it used a non-proportional †1 font (Lucida console):

The advantage of the non-proportional font was that it was easy to make lists (e.g. lists of parameters) that all lined up correctly. This would be less of a problem in TIA Portal if the comment fields accepted the tab character, but tabs are not permitted in comment fields — they are ignored.

Comment fields do not support images of any sort or format.

†1 Proportional fonts have letters that are different widths, the M character is wider than the I character. With non-proportional fonts all characters are the same width (like text written on a typewriter).

These restrictions can make TIA Portal comments look untidy:

Here (Figure 3.3) everything is lined up using spaces, the discrepancies are most clearly seen with the Ms, they don’t line up properly.

Things can be improved slightly by using smaller whitespace characters. The smallest such character is 1/6 of an em space. A normal space is 1/4 of an em space, the 1/6 em space has been used to better line up the Ms (Figure 3.4):

The lining is better but not perfect. Again it shows up with the Ms largely because the Ms all have a vertical leading edge and the eye is sensitive to the misalignment. Where such lists are used, it is better to have the column with similar entries as the first column on the left, these always line up (Figure 3.5):

In this example the different characters on each line in columns 2 and 3 breakup the straight line and give the table a cleaner appearance (the actual alignment is no better than that in Figure 3.4, it just looks better).

The Siemens TIA Portal Basic font supports the following whitespace characters (given in relation to both the em space and the normal space):

The table shows the relative size of the whitespace as the gap between the two blocks. Other fonts support smaller spaces (thin-space, hair-space); these are also available to the TIA Portal comment fields, but are set to the same spacing as the characters listed above. The alt codes†2 listed here are for use in Word, they don’t work properly in TIA Portal.

Generally, it is better to prepare comments in Word and then paste them into TIA Portal (TIA Portal does not have a spelling checker).

The TIA Portal comment fields are important and should contain a full description of the fundamental functions and properties of the block in question. This will inevitably involve tables and lists and some effort and time should be taken in making these look presentable (there is further discussion about the use of whitespace to achieve this § xxx).

†2 Alt codes are a short cut for entering symbols into the text of a document; they are referenced throughout this document like so: alt + 9674, this will insert the lozenge character (◊).
Alt codes are entered by holding down the alt key and entering the specified digits on the numeric keypad (note: num lock must be on). All digits must be entered, even a leading zero. There is no time limit; the character is inserted when the alt key is released.

3.1.1

User Documentation

TIA Portal also has a more powerful User Documentation mechanism, this supports Word documents, text files (txt and rtf), Excel spread sheets, XPS files, html and pdfs.

The User Documentation is embedded within the TIA Project and is accessible from the program package simply by selecting the relevant object (blocks &c.) and pressing shift+f1 (a consent box will open the first time this is done).

User Documentation is the main mechanism for documenting the PAL blocks. However, the comments within the block are necessary and must be detailed enough to provide a full description of how the block works, how to use it and a full explanation of all the parameters and data structures used by the block.

Common headers and networks

Each programmable block within the PAL (FB, FC or OB) has several common areas at the start of the block:

3.2.1

Block title and comment field

The first item in any programmable block (FB, FC or OB) is the block title field with a comment area underneath, highlighted in Figure 3.6:

The block title line must contain an appropriate title for the block; this is generally given in the software module design specification (SMDS; this document is the SMDS for FC 00001) for the block as document property 003 BlkTitle. In the case of this block it is Template for Block Documentation and Styles.

Generally, the block title should be no more than 100 characters (including spaces) and should be in title case (i.e. all principle†3 words are capitalised).

†3 Principal word are words that are not:
  ▪ Articles (a, an, the)
  ▪ Conjunctions (and, or, but &c.)
  ▪ Prepositions (in, with, on &c.)

The comment field contains a standard copyright message, details of how to access the specific user documentation for the block and a link to the full PAL documentation website. It is shown below in Figure 3.7:

The comment field is common to all blocks that form part of the PAL, this includes the standard blocks (that hold fixed code for a particular function) and template blocks (containing examples that indicate how the software should be structured).

The comment field is exactly 4 lines long, each line ending with a paragraph delimiter—¶ (i.e. has received a carriage return, it can be seen in Word if the show/hide button — home tab, paragraph group, ¶ button is active).

The reason for limiting the block title comment field to 4 lines is that they show up as the comment text in the block properties window:

The block properties tab generally shows the first four lines of the block title comment field (if the screen is wide enough). Each line is started with two heavy horizontal lines (alt + 9473); this is to clearly delineate the start of the line.

  5. Block calls

  6. Associated blocks

  7. System block calls and system data types

The description in the block is complete and holds full examples of all the different types of tables used within the various sections.

3.2.3

Network 2 — Current revision and modification history

Network 2 contains the current revision of the software module. This network is not empty, it contains the current revision number of the block, the revision date and the author’s initials. These are hard coded (using statement list) in the network.

Network 2 always has the title: Current revision and modification history. The network comment field contains the modification history of the block (up to and including the current revision).

The comment field has the following format:

The code associated with the network has the following format:

Network 2
      L     0.01                 // Current revision number
      L     D#2018-05-30         // Revision date
      L     'MG'                 // Author
Code 3.1 — Network 2 - current revision and modification history

All blocks within the PAL (with the exception of OB 1, see § 3.3.2) have the block revision number stored in Network 2. The current revision is stored as a real (floating point) number with two decimal places.

The revision date is stored as a date. type. This has the format D#YYYY-MM-DD this is the IEC (International Electrotechnical Commission) standard format for dates.

The author’s initials are stored as a four (maximum) character string.

3.3

OB 1 header and revision networks

OB 1 (the main organisation block) is considered a special organisation block in terms of the Practical Series Automation Library (and in terms of most Siemens controller software). It is the block that executes all the rest of the controller software.

As such it contains information about the whole project rather than just a software module. The revision data is also project specific (not module specific).

3.3.2

OB 1 Network 2 — Current revision and modification history

OB 1 Network 2 contains the current revision of the whole software project (rather than of a particular block). The whole project is issued at a particular release and this is independent of individual module revisions — most modules do not change their revision when the project as a whole is reissued at a new release, the new release may be to include new blocks or it may be to include a change or correction to a particular block (in which case the revision of the affected block would change, but not the others).

The overall software is version controlled and receives various different revision numbers throughout the development, test, and deployment stages of the project. The software version contains both a release status and revision number in the form:

Xnn.nn

Where X is a letter indicating the status of the software and nn.nn is the revision number. The status can have the following values:

Release Meaning Description
D Development Code is under development and has not been tested
T Test Code is released for testing
C Commissioning Software is deployed to site and is being commissioned
Q Qualification Software is released for qualification (IQ/OQ)
R Released Released for use
Table 3.2   Software release details

The software revision number also reflects the status of the software; revision numbers always increase at each software revision and specific number ranges have different meanings as follows:

Rev Range Meaning Description
00.0-00.99 Pre-test development Code is under development and has not been tested
01.00-01.99 Test stage Test revisions of the software
02.00-02.99 Site deployment Software is deployed to site and is being commissioned
03.00-99.00 Released Software is released for use and is under customer control
Table 3.3   Software revision number

The software revision number is the unique reference for the software, the release state is indicative only (it would be possible for tested software to drop back into development, its release would change back to D to indicate this, but its revision number would stay in the 01.nn range (revision numbers never decrease in value).

Generally, in the development stages, the software is largely uncontrolled (and is certainly untested), incremental revisions take place frequently at this stage (often daily or when a significant amount of work has been done) and these revisions cause incremental increases in the two decimal places of the revision number (i.e. 00.nn). If there is likely to be more than 99 development revisions, then the number of decimal places can be increased (e.g. 00.nnn).

All this information is stored in Network 2 of OB 1.

Like all other software blocks, Network 2 in OB 1 has the title: Current revision and modification history. The network comment field contains the modification history of the software project as a whole (up to and including the current revision).

Code 3.2 — Network 2 - OB1 current revision and modification history

OB 1 stores more revision information than is done with the general software block revision data (see § 3.2.3) and it stores it in a slightly different way.

In all other blocks, the revision, the revision date and author’s initials are hard coded as load statements that simply transfer the data to the CPU accumulators (it doesn’t actually do anything with the data, it is not transferred anywhere else, and it will simply be overwritten the next time the CPU loads its accumulators).

In OB 1 the data is loaded in the same way, but this time it is transferred to the system data block (DB21000_StdSysData_D). This allows the data to be read elsewhere within the software or by an external device (such as a SCADA or HMI system). In short, OB 1 makes the current revision of the project available to anything that has access to the controller.

In addition to the revision number, revision date and author’s initials, OB 1 also stores the status of the software (Table 3.1) as a char data type.

The other entries are in the same format as that for any other block: the current revision is stored as a real (floating point) number with two decimal places.

The revision date is stored as a date type. This has the format D#YYYY-MM-DD, again this is the IEC (International Electrotechnical Commission) standard format for dates.

The author’s initials are stored as a four (maximum) character string.

AUTHOR’S NOTE

Bet you’re impressed with the figures in Network 7 — they took a bit of time.

Thing is, it’s fairly easy — the lines and boxes are all created with the box drawing characters and these are all exactly 1 em-space wide.

The problem comes when non-box characters, such as letters, are used. Under these circumstances other whitespace characters must be used to pad the gaps out to a whole em-space.

The alphanumeric characters all have varying widths; however, and this is just a lucky feature of the font, they are all either a whole multiple of the normal space character (¼ em) or a multiple of the normal space character plus ⅓ of a normal space, or a multiple of the normal space character plus ⅔ of a normal space.

This is fortunate because the extra whitespace characters available in TIA Portal have a width of ⅔ of a normal space (SIX-PER-EM space) or 1⅓ of a normal space (THREE-PER-EM space) — see Table 3.1.

The upshot of this is that with judicious use of the NORMAL SPACE, THREE-PER-EM space and the SIX-PER-EM space it is possible to always get a perfect fit.

It just needs a bit of experimentation; it’s either a space, a space and a SIX-PER-EM or just a THREE-PER-EM.

Michael Gledhill

The .git folder—a golden rule

The .git folder is a hidden folder in the root directory of the repository. It contains all that is important: the local repository, any staged files, all the metadata associated with the repository (change records, logs &c.) and all the important bits for a tracked project.

There is one golden rule concerning the .git folder
DON’T MESS WITH IT

Best thing is, don’t even look inside it. If you delete it, you delete everything,
if you change it, everything gets screwed up
